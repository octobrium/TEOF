# Frameworks Layer

This layer contains domain-specific frameworks that integrate existing, reality-tested approaches with TEOF's selection and audit functions.

## Core Principle: Curate, Don't Create

**TEOF is the glue, not the content.**

Rather than deriving domain principles from first principles, this layer:
1. **Selects** existing frameworks that have survived reality-testing
2. **Analyzes** why they persist in their domains
3. **Integrates** them into a coherent personal system
4. **Audits** outputs for contradictions and coherence

## Structure

Each subdirectory addresses a life domain (ordered by TEOF layer hierarchy):

- **health/** - Body, energy, sleep, exercise, nutrition (Layer 2: Energy)
- **social/** - People, network, relationships, Twitter, boundaries (Layer 3: Propagation)
- **finances/** - Money, income, investments, defense (Layer 4: Defense)
- **power/** - Systems, institutions, navigation, leverage (Layer 7: Power)
- **work/** - Career, projects, skills, professional growth (derived)
- **code/** - Software development practices (derived)

## Two Types of Content

### 1. Source Framework Analysis
For each domain, analyze existing frameworks that have persisted:

| Domain | Source Frameworks | Track Record |
|--------|-------------------|--------------|
| Discipline | Islamic practices (fasting, prayer rhythm) | 1400 years |
| Community | Christian/Islamic congregation models | 1400-2000 years |
| Knowledge | Scientific method, empiricism | 400 years |
| Code | Unix philosophy, Clean Code, SOLID | 20-50 years |
| Finance | Value investing (Graham/Buffett) | 80+ years |
| Combat | Martial traditions (BJJ, boxing) | Decades-centuries |

Analysis should cover:
- What makes it robust?
- What selection pressures has it survived?
- What's core vs peripheral?
- What observable outcomes does it produce?

### 2. Integrated Frameworks
Personal frameworks that combine the best of existing approaches:

- May borrow from multiple source frameworks
- Should explicitly attribute borrowed elements
- Should note where sources conflict and how conflicts are resolved
- Should be treated as living documents refined through use

## How to Use

### For Existing Frameworks
Each framework doc should:
1. Identify source frameworks being borrowed from
2. Explain why those sources were selected (track record, outcomes)
3. Note what's adapted vs used directly
4. Identify where sources conflict and resolution approach
5. Provide actionable guidance

### For Adding Source Analysis
When analyzing why an existing framework persists:
1. What domain does it address?
2. How long has it survived? Under what pressures?
3. What observable outcomes do practitioners show?
4. What's the core (stable) vs periphery (adaptive)?
5. Where might it integrate or conflict with other selected frameworks?

### For Creating New Integrated Frameworks
When combining sources for a new domain:
1. Inventory what existing frameworks address this domain
2. Select based on track record and observable outcomes
3. Map where they agree, conflict, or have gaps
4. Integrate with explicit attribution
5. Note TEOF's role: selection, integration, audit—not content generation

## Pattern C Role

This is the **Frameworks** layer:
- **Core:** The source frameworks themselves (centuries of refinement)
- **Operational:** TEOF's selection criteria and integration method
- **Tactical:** Personal adaptations and context-specific applications

## TEOF's Function Here

TEOF provides:
1. **Selection criteria:** What has persisted? What outcomes are observable?
2. **Integration method:** How to combine without contradiction?
3. **Audit function:** Is the combined output coherent?
4. **Nothing else:** TEOF doesn't replace domain expertise
